# Disaster Recovery Runbook

Note: The August 1, 2026 launch plan defers Actions Runner Controller. Recovery
should not assume ARC is available. Use the manual Helm deployment flow in
`docs/production-deployment-plan.md` unless ARC is explicitly reintroduced after
launch.

This runbook describes how to rebuild the AlMadar platform from source control
and backups. It assumes the engineer performing recovery has no prior project
context beyond this repository, access to the approved secret store, and
emergency access to OCI, Cloudflare, and GitHub.

The platform is recoverable only if these external backups exist:

- Terraform remote state backups or a known-good exported state snapshot.
- OCI PostgreSQL backups for each environment.
- Object Storage bucket backups or replicated bucket contents.
- OCI Vault backup/export procedure for required secret payloads, or an
  approved break-glass secret source.
- GitHub repository access, including Actions variables and secrets.

Do not perform destructive commands during recovery unless the incident
commander explicitly approves them.

## Recovery Objectives

Target recovery objectives must be confirmed with project leadership for each
incident.

| Component | Initial target | Notes |
| --- | --- | --- |
| Public frontend | Restore first if content can temporarily be stale | Depends on OKE and Strapi/API health |
| Strapi CMS | Restore after database and object storage | Content model changes require migration success |
| IIIF image delivery | Restore after object storage and Cantaloupe | Cached Cloudflare content may mask origin issues |
| GitHub Actions runners | Restore after OKE | Needed for normal deployments |

## Contacts And Access

Before starting recovery, confirm access to:

- OCI tenancy and target compartment.
- OCI CLI profile with permission for Networking, OKE, PostgreSQL, Object
  Storage, Vault, IAM, and Load Balancer.
- Terraform backend credentials.
- Kubernetes kubeconfig, if the cluster still exists.
- Cloudflare zone admin.
- GitHub organization admin or repository admin.
- OCIR credentials.

## Repository Bootstrap

Start every recovery from a clean clone.

```bash
git clone <repository-url> almadar-platform-terraform
cd almadar-platform-terraform
export REPO_ROOT="$(pwd)"
git status
```

Install local tools:

```bash
terraform version
oci --version
kubectl version --client
helm version
docker version
```

If any command is missing, install it before proceeding.

## Terraform Recovery Principles

Terraform roots are intentionally separated. Rebuild them in this order:

1. Network: `infrastructure/terraform/environments/network/<env>`
2. Object Storage: `infrastructure/terraform/environments/object-storage`
3. PostgreSQL: `infrastructure/terraform/environments/postgresql`
4. OKE: `infrastructure/terraform/environments/oke`
5. OKE namespaces and RBAC: `infrastructure/terraform/environments/oke-rbac`
6. Vault secrets: `infrastructure/terraform/environments/vault`
7. Kubernetes manifests and Helm releases.

Use `terraform plan` before every `terraform apply`.

```bash
cd <terraform-root>
cp terraform.tfvars.example terraform.tfvars
vi terraform.tfvars
terraform init
terraform plan
terraform apply
```

If remote Terraform state is available, `terraform init` should attach to it.
If state is missing but OCI resources still exist, do not recreate duplicates.
Either restore the backend state backup or import existing resources into state.

State inspection commands:

```bash
terraform state list
terraform state pull > recovered.tfstate
terraform output
terraform output -json
```

Import is a last resort. Use it only after identifying the exact OCI resource
OCID and matching Terraform resource address.

```bash
terraform import '<resource-address>' '<oci-resource-ocid>'
terraform plan
```

## Scenario 1: Lost Kubernetes Cluster

Use this when the OKE cluster is deleted, unrecoverable, or replaced.

### Impact

- Application pods are gone.
- GitHub Actions self-hosted runners are gone.
- Kubernetes Secrets generated by External Secrets are gone.
- OCI PostgreSQL and Object Storage may still be intact.

### Recovery Steps

1. Confirm the network stack still exists.

   ```bash
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/network/${ENV} init
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/network/${ENV} plan
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/network/${ENV} output -json > /tmp/almadar-network-${ENV}.json
   ```

2. Recreate OKE.

   ```bash
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/oke init
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/oke plan
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/oke apply
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/oke output kubectl_command
   ```

3. Verify cluster access.

   ```bash
   kubectl --kubeconfig ${REPO_ROOT}/infrastructure/terraform/environments/oke/generated/oke-kubeconfig.yaml get nodes
   ```

4. Recreate namespaces and RBAC.

   ```bash
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/oke-rbac init
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/oke-rbac plan
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/oke-rbac apply
   kubectl --kubeconfig ${REPO_ROOT}/infrastructure/terraform/environments/oke/generated/oke-kubeconfig.yaml get namespaces dev test prod
   ```

5. Reinstall External Secrets Operator.

   The operator installation method must match the approved platform standard.
   After the operator is installed, apply this repo's manifests:

   ```bash
   kubectl apply -k ${REPO_ROOT}/infrastructure/kubernetes/external-secrets
   kubectl -n external-secrets get pods
   ```

6. Recreate Kubernetes application secrets from OCI Vault.

   ```bash
   kubectl -n dev get externalsecret almadar-secrets
   kubectl -n test get externalsecret almadar-secrets
   kubectl -n prod get externalsecret almadar-secrets
   kubectl -n ${ENV} get secret almadar-secrets
   ```

7. Reinstall GitHub Actions Runner Controller.

   ```bash
   kubectl apply -k ${REPO_ROOT}/infrastructure/kubernetes/actions-runner-controller
   kubectl -n arc-runners wait externalsecret/arc-github-app --for=condition=Ready --timeout=120s

   helm upgrade --install arc \
     oci://ghcr.io/actions/actions-runner-controller-charts/gha-runner-scale-set-controller \
     --namespace arc-system \
     --values ${REPO_ROOT}/infrastructure/helm/actions-runner-controller/controller-values.yaml

   helm upgrade --install almadar-oci-runners \
     oci://ghcr.io/actions/actions-runner-controller-charts/gha-runner-scale-set \
     --namespace arc-runners \
     --values ${REPO_ROOT}/infrastructure/helm/actions-runner-controller/runner-scale-set-values.yaml \
     --set githubConfigUrl=https://github.com/<GITHUB_ORG>
   ```

8. Redeploy application Helm releases.

   Replace image repositories and tags with the last known good images from
   OCIR or the GitHub Actions deployment history.

   ```bash
   helm upgrade --install almadar-strapi ${REPO_ROOT}/infrastructure/helm/strapi \
     --namespace ${ENV} \
     --values ${REPO_ROOT}/infrastructure/helm/strapi/values-${ENV}.yaml \
     --values ${REPO_ROOT}/infrastructure/helm/strapi/values-ci.yaml \
     --set image.repository=<ocir>/<namespace>/almadar-strapi \
     --set image.tag=<known_good_tag> \
     --wait \
     --atomic \
     --timeout 10m

   helm upgrade --install almadar-frontend ${REPO_ROOT}/infrastructure/helm/frontend \
     --namespace ${ENV} \
     --values ${REPO_ROOT}/infrastructure/helm/frontend/values-${ENV}.yaml \
     --set image.repository=<ocir>/<namespace>/almadar-frontend \
     --set image.tag=<known_good_tag> \
     --wait \
     --atomic \
     --timeout 10m

   helm upgrade --install almadar-cantaloupe ${REPO_ROOT}/infrastructure/helm/cantaloupe \
     --namespace ${ENV} \
     --values ${REPO_ROOT}/infrastructure/helm/cantaloupe/values-${ENV}.yaml \
     --wait \
     --atomic \
     --timeout 10m
   ```

9. Update Cloudflare DNS if the OCI Load Balancer hostname changed.

   See `docs/cloudflare.md`.

10. Verify service health.

   ```bash
   kubectl -n ${ENV} get pods
   kubectl -n ${ENV} get svc
   kubectl -n ${ENV} rollout status deployment/almadar-strapi --timeout=5m
   kubectl -n ${ENV} rollout status deployment/almadar-frontend --timeout=5m
   kubectl -n ${ENV} rollout status deployment/almadar-cantaloupe --timeout=5m
   curl -I https://www.${DOMAIN}
   curl -I https://cms.${DOMAIN}/admin
   curl -I https://api.${DOMAIN}/api
   curl -I https://iiif.${DOMAIN}/
   ```

### Acceptance

- Nodes are Ready.
- `dev`, `test`, and `prod` namespaces exist.
- External Secrets reconciles `almadar-secrets`.
- GitHub Actions runner smoke test runs on OCI.
- Frontend, Strapi, and Cantaloupe rollouts complete.
- Cloudflare routes traffic to the new load balancer.

## Scenario 2: Lost PostgreSQL Database

Use this when the OCI PostgreSQL DB system is deleted, corrupted, or must be
restored to an earlier backup.

### Impact

- Strapi content and configuration are unavailable or stale.
- Content model migration state may be lost.
- Frontend API-backed pages may fail.

### Recovery Steps

1. Stop Strapi writes.

   ```bash
   kubectl -n ${ENV} scale deployment/almadar-strapi --replicas=0
   ```

2. Identify the backup to restore.

   ```bash
   oci psql backup list \
     --compartment-id ${OCI_COMPARTMENT_OCID} \
     --region ${OCI_REGION}
   ```

3. Restore the DB system from backup.

   Use the backup OCID selected by the incident commander.
   Confirm exact restore options for the installed OCI CLI before running the
   restore:

   ```bash
   oci psql db-system restore --help
   ```

   ```bash
   export PSQL_BACKUP_OCID=<backup_ocid>
   export PSQL_DB_SYSTEM_OCID=<db_system_ocid>

   oci psql db-system restore \
     --db-system-id ${PSQL_DB_SYSTEM_OCID} \
     --backup-id ${PSQL_BACKUP_OCID} \
     --region ${OCI_REGION}
   ```

   If the restore creates a replacement DB system instead of restoring in
   place, capture its endpoint and update Terraform variables and Vault
   payloads before restarting Strapi.

4. Reconcile Terraform state.

   If the DB system OCID changed, update the PostgreSQL stack or import the
   restored DB resource according to the actual resource state.

   ```bash
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/postgresql init
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/postgresql plan
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/postgresql import '<resource-address>' '<new_db_system_ocid>'
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/postgresql plan
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/postgresql output -json strapi_environment
   ```

5. Update OCI Vault Strapi database secrets.

   ```bash
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/vault init
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/vault plan
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/vault apply
   ```

6. Force External Secrets refresh if credentials changed.

   ```bash
   kubectl -n ${ENV} annotate externalsecret almadar-secrets \
     force-sync="$(date +%s)" --overwrite
   kubectl -n ${ENV} get secret almadar-secrets
   ```

7. Run Strapi migration and restart.

   ```bash
   helm upgrade --install almadar-strapi ${REPO_ROOT}/infrastructure/helm/strapi \
     --namespace ${ENV} \
     --values ${REPO_ROOT}/infrastructure/helm/strapi/values-${ENV}.yaml \
     --values ${REPO_ROOT}/infrastructure/helm/strapi/values-ci.yaml \
     --set image.repository=<ocir>/<namespace>/almadar-strapi \
     --set image.tag=<known_good_tag> \
     --set migrations.enabled=true \
     --wait \
     --atomic \
     --timeout 10m
   ```

8. Verify content and admin access.

   ```bash
   kubectl -n ${ENV} logs deployment/almadar-strapi --tail=200
   curl -I https://cms.${DOMAIN}/admin
   curl -I https://api.${DOMAIN}/api
   ```

### Acceptance

- PostgreSQL is available and healthy.
- Strapi migration job succeeds.
- Strapi admin loads.
- Public API returns expected content.
- The restored backup timestamp is recorded in the incident notes.

## Scenario 3: Lost Object Storage Bucket

Use this when an IIIF or Strapi media bucket is deleted, emptied, corrupted, or
must be restored from backup.

### Impact

- IIIF images may be missing.
- Strapi uploads may be missing.
- Cloudflare may continue serving cached copies temporarily.

### Recovery Steps

1. Identify affected bucket.

   ```bash
   export BUCKET=iiif-${ENV}
   # or
   export BUCKET=strapi-${ENV}
   ```

2. Recreate buckets and IAM if needed.

   ```bash
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/object-storage init
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/object-storage plan
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/object-storage apply
   ```

3. Restore objects from backup.

   From a local backup directory:

   ```bash
   export RESTORE_DIR=/path/to/restored/${BUCKET}

   oci os object bulk-upload \
     --namespace-name ${OCI_OBJECT_NAMESPACE} \
     --bucket-name ${BUCKET} \
     --src-dir ${RESTORE_DIR} \
     --region ${OCI_REGION}
   ```

   From another OCI bucket or staged directory, sync changed files:

   ```bash
   oci os object sync \
     --namespace-name ${OCI_OBJECT_NAMESPACE} \
     --bucket-name ${BUCKET} \
     --src-dir ${RESTORE_DIR} \
     --region ${OCI_REGION}
   ```

4. Verify object count and sample reads.

   ```bash
   oci os object list \
     --namespace-name ${OCI_OBJECT_NAMESPACE} \
     --bucket-name ${BUCKET} \
     --region ${OCI_REGION} \
     --limit 10
   ```

5. Reconcile Vault S3 credentials if bucket names or credentials changed.

   ```bash
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/vault init
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/vault plan
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/vault apply
   kubectl -n ${ENV} annotate externalsecret almadar-secrets \
     force-sync="$(date +%s)" --overwrite
   ```

6. Restart affected services.

   ```bash
   kubectl -n ${ENV} rollout restart deployment/almadar-strapi
   kubectl -n ${ENV} rollout restart deployment/almadar-cantaloupe
   kubectl -n ${ENV} rollout status deployment/almadar-strapi --timeout=5m
   kubectl -n ${ENV} rollout status deployment/almadar-cantaloupe --timeout=5m
   ```

7. Purge affected Cloudflare cache.

   See `docs/cloudflare.md`. Prefer targeted purge for affected IIIF or media
   URL prefixes.

### Acceptance

- Bucket exists with versioning and lifecycle policy.
- Restored objects are listed in OCI.
- Strapi uploaded media resolves.
- IIIF sample image resolves through Cantaloupe.
- Cloudflare no longer serves stale missing-object responses.

## Scenario 4: Failed Deployment

Use this when a GitHub Actions deployment, migration, or Helm rollout fails.

### Impact

- New release may be partially deployed.
- Strapi migrations may have failed before rollout.
- Previous release should remain active if Helm `--atomic` succeeded.

### Recovery Steps

1. Identify failing environment and release.

   ```bash
   export ENV=dev
   helm -n ${ENV} list
   helm -n ${ENV} history almadar-strapi
   helm -n ${ENV} status almadar-strapi
   ```

2. Inspect Kubernetes events and pods.

   ```bash
   kubectl -n ${ENV} get pods
   kubectl -n ${ENV} describe pod <pod_name>
   kubectl -n ${ENV} get events --sort-by=.lastTimestamp
   ```

3. Inspect migration job, if present.

   ```bash
   kubectl -n ${ENV} get jobs
   kubectl -n ${ENV} logs job/almadar-strapi-migrate
   ```

4. Roll back the Helm release if the current revision is bad.

   ```bash
   helm -n ${ENV} history almadar-strapi
   helm -n ${ENV} rollback almadar-strapi <previous_good_revision> --wait --timeout 10m
   kubectl -n ${ENV} rollout status deployment/almadar-strapi --timeout=5m
   ```

5. If the failed release changed the database, determine whether rollback is
   safe.

   - If migrations did not run, redeploy the previous image.
   - If migrations ran and are backward compatible, roll back the app.
   - If migrations ran and are not backward compatible, restore PostgreSQL from
     backup using Scenario 2.

6. Redeploy a known-good image.

   ```bash
   helm upgrade --install almadar-strapi ${REPO_ROOT}/infrastructure/helm/strapi \
     --namespace ${ENV} \
     --values ${REPO_ROOT}/infrastructure/helm/strapi/values-${ENV}.yaml \
     --values ${REPO_ROOT}/infrastructure/helm/strapi/values-ci.yaml \
     --set image.repository=<ocir>/<namespace>/almadar-strapi \
     --set image.tag=<known_good_tag> \
     --wait \
     --atomic \
     --timeout 10m
   ```

7. Re-run GitHub Actions only after the root cause is fixed.

   ```text
   .github/workflows/strapi.yml
   ```

### Acceptance

- Helm release is deployed at a known-good revision.
- Pods are Ready.
- Strapi admin and API work.
- Any database restore decision is recorded.
- Failed deployment root cause is documented before retry.

## Scenario 5: Region Outage

Use this when the primary OCI region is unavailable or expected to remain
unavailable longer than the accepted outage window.

### Impact

- Primary OKE, PostgreSQL, Object Storage, and Load Balancer may be unavailable.
- Cloudflare can reroute DNS only after a standby origin exists.
- Recovery depends on replicated backups in the secondary region.

### Recovery Region

Current planned OCI regions:

```text
Primary: me-riyadh-1
Secondary: me-jeddah-1
```

Confirm the target region with the incident commander before applying
Terraform.

### Recovery Steps

1. Freeze deployments.

   Disable or pause GitHub Actions deployments until the recovery region is
   stable.

2. Set recovery variables.

   ```bash
   export ENV=prod
   export OCI_REGION=me-jeddah-1
   export DOMAIN=<domain>
   ```

3. Rebuild regional network.

   ```bash
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/network/prod init
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/network/prod plan -var='region=me-jeddah-1'
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/network/prod apply -var='region=me-jeddah-1'
   ```

   If region is not currently a variable in that stack, create a recovery
   `terraform.tfvars` from the example and set the region there.

4. Rebuild Object Storage buckets in the recovery region.

   ```bash
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/object-storage init
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/object-storage plan
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/object-storage apply
   ```

5. Restore object backups into recovery buckets.

   Use Scenario 3 for each required bucket:

   ```text
   iiif-prod
   strapi-prod
   ```

6. Restore PostgreSQL in the recovery region.

   Use Scenario 2 with a backup replicated or exported to the recovery region.

7. Recreate OKE and RBAC.

   ```bash
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/oke init
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/oke plan
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/oke apply

   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/oke-rbac init
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/oke-rbac plan
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/oke-rbac apply
   ```

8. Recreate Vault and External Secrets configuration.

   ```bash
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/vault init
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/vault plan
   terraform -chdir=${REPO_ROOT}/infrastructure/terraform/environments/vault apply

   kubectl apply -k ${REPO_ROOT}/infrastructure/kubernetes/external-secrets
   kubectl -n prod wait externalsecret/almadar-secrets --for=condition=Ready --timeout=120s
   ```

9. Deploy applications and ARC.

   Use Scenario 1 steps for ARC and application Helm releases.

10. Cut Cloudflare over to the recovery origin.

   Update `www`, `cms`, `api`, and `iiif` DNS targets or Cloudflare Load
   Balancer origins to point to the recovery OCI Load Balancer.

   See `docs/cloudflare.md`.

11. Validate production.

   ```bash
   curl -I https://www.${DOMAIN}
   curl -I https://cms.${DOMAIN}/admin
   curl -I https://api.${DOMAIN}/api
   curl -I https://iiif.${DOMAIN}/
   ```

### Acceptance

- Recovery region serves all production hostnames.
- PostgreSQL restore point is known.
- Object Storage restore point is known.
- GitHub Actions deployments are re-enabled against the recovery cluster.
- Cloudflare points to the recovery OCI Load Balancer.

## Backup Requirements

The following backup controls must exist before the platform is considered
production-ready:

- Scheduled OCI PostgreSQL backups with restore testing.
- Object Storage replication or scheduled exports for `iiif-*` and `strapi-*`.
- Remote encrypted Terraform state with versioning.
- Documented export or break-glass recovery path for OCI Vault secret payloads.
- OCIR image retention for last known good releases.
- GitHub Actions variables and secrets recovery procedure.
- Cloudflare configuration exported to IaC or documented after each change.

## Recovery Drill

Run a non-production recovery drill at least once per release cycle until the
procedures are proven.

Minimum drill:

1. Rebuild `dev` OKE from Terraform.
2. Restore `dev` Strapi database from backup.
3. Restore one IIIF object prefix into `iiif-dev`.
4. Deploy Strapi, frontend, and Cantaloupe.
5. Run GitHub Actions runner smoke test.
6. Record elapsed time, failures, and documentation gaps.
